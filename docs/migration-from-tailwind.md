# Migration From Tailwind

Synced Flow is not a one-for-one Tailwind replacement. The useful migration
path is to keep class names complete, move project identity into tokens, and
replace breakpoint-heavy layout with fluid primitives over time.

## Package Setup

Remove Tailwind packages when the project no longer needs them:

```bash
pnpm remove tailwindcss @tailwindcss/postcss @tailwindcss/vite tailwind-merge
pnpm add @synced/flow
pnpm exec synced-flow init
```

## Class Migration

Start with compatibility utilities generated by `synced-flow build`, then move
repeated layout patterns to Synced Flow primitives.

| Tailwind-style pattern | Synced Flow direction |
| --- | --- |
| `container mx-auto px-6` | `sf-container` |
| `py-16 sm:py-24` | `sf-section` or fluid space tokens |
| `flex flex-col gap-4` | `sf-stack` |
| `flex flex-wrap gap-3` | `sf-cluster` |
| `grid gap-6 md:grid-cols-3` | `sf-auto-grid` or project CSS with `minmax()` |
| `bg-primary text-primary-foreground` | Keep semantic utility classes |

## Responsive Variants

Use `responsiveVariants: true` only while migrating old code:

```js
export default defineConfig({
  scan: ['app', 'components', 'lib'],
  out: 'app/synced-flow.generated.css',
  responsiveVariants: true,
})
```

Turn it off for new projects and once layout has moved to fluid primitives.

## Dynamic Classes

Avoid building class names from fragments.

```tsx
// Good
const states = {
  open: 'block opacity-100',
  closed: 'hidden opacity-0',
}

// Avoid
const className = `${isOpen ? 'block' : 'hidden'} opacity-${amount}`
```

Use `safelist` when a dynamic class is unavoidable.
